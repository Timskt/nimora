# Nimora 离线能力与数据生命周期规范

> 版本：0.1.0-draft  
> 更新日期：2026-07-17

## 1. 离线承诺

无网络、无登录、无 AI Key、Registry 不可用时，以下能力必须可用：

- Pet Runtime、默认角色和已安装资源。
- 本地 Command、Profile、养成、成就和记忆管理。
- 本地 Automation、提醒和系统通知。
- 已安装且不依赖远程服务的 Skill。
- 本地 Gateway 和回环地址 Connector。
- 配置导入导出、诊断和安全模式。

网络不是启动前置条件。应用启动不得等待更新、商店、遥测或 AI Provider。

## 2. 能力降级标记

每个 Skill、Connector 和 Agent Pack 必须声明：

```json
{
  "offline": "full | partial | unavailable",
  "cachePolicy": "none | metadata | content",
  "requiresNetworkFor": ["calendar.sync"],
  "offlineFallback": "use-last-cache"
}
```

UI 必须区分“离线不可用”“权限不足”“服务故障”和“尚未配置”。

## 3. 本地优先同步模型

- 本地数据库是桌面运行时的即时事实来源。
- 云同步是可选复制层，不是 Core 数据库。
- 每条可同步记录包含稳定 ID、版本、更新时间、设备 ID 和删除墓碑。
- 冲突策略按数据类型定义，不能全局使用最后写入获胜。
- 密钥、原始审计、活动窗口和 Restricted 数据默认不参与同步。

## 4. 数据目录

```text
data/       数据库和用户状态
config/     可导出配置和 Profile
packages/   已安装包和 lockfile
cache/      可重建缓存
logs/       轮转运行日志
audit/      受控审计
backups/    版本化本地快照
```

实际路径遵循平台规范。缓存删除不得影响应用启动和用户内容。

## 5. 数据所有权与分类

| 数据 | 所有者 | 默认保留 | 导出 |
|---|---|---|---|
| 宠物状态/关系 | 用户 | 直到删除 | 是 |
| 配置/Profile | 用户 | 直到删除 | 是 |
| Agent 记忆 | 用户 | 可配置 | 是 |
| 扩展数据 | 用户/扩展命名空间 | 卸载时询问 | 是 |
| 审计 | 用户 | 默认 30 天 | 是 |
| 运行日志 | 系统 | 默认 14 天 | 诊断导出 |
| 缓存 | 系统 | 按容量淘汰 | 否 |

## 6. 导入导出

导出包必须包含 manifest、Schema 版本、应用版本、创建时间、数据分类和文件 hash。默认不包含密钥、Restricted 正文、设备标识和不可转让商业资源。

导入流程：验证 → 预览 → 冲突策略 → 快照 → 事务应用 → 健康检查 → 确认或回滚。

## 7. 删除语义

- 用户删除必须说明本地、云端和备份影响范围。
- 敏感数据支持立即删除；备份中的删除遵循明确保留周期。
- 扩展卸载提供“保留设置”和“删除全部数据”。
- 记忆删除后不得继续进入 Agent 上下文、搜索索引和同步队列。

## 8. 当前 Schema

- 所有持久化根对象包含 `schemaVersion`。
- `schemaVersion` 是当前数据契约的类型标识，不代表项目已经存在历史版本或迁移链。
- 首个稳定版前只维护唯一当前 Schema，不为开发期中间结构保留迁移代码。
- 开发期结构变化直接重建当前 Schema 和测试数据，不实现迁移、降级或双写。
- 稳定版发布后若真实已发布数据需要升级，再单独设计幂等迁移、快照和磁盘空间检查。
- 新版本不认识的字段应保留或显式拒绝，不能静默丢失。
- 当前实现明确拒绝未知 Schema；数据库无法验证时进入隔离恢复模式，不尝试迁移、修复或覆盖原文件。

## 9. 缓存策略

- 商店元数据可缓存并标记更新时间。
- 远程图片与资源按 hash 缓存，验证类型和大小。
- AI 回复不作为离线知识缓存自动复用。
- 日历、天气等时效数据必须显示数据时间。
- 缓存淘汰不能删除已安装包当前使用版本。

## 10. 离线测试

- 首次启动后断网，重启应用并完成核心互动。
- 断网期间创建规则、切换资源、导出配置。
- 网络恢复后只恢复允许同步的任务，不重复执行过期动作。
- Registry 不可用时已安装包可启动，更新操作给出明确状态。
- Provider 断网时 Agent 失败不影响 Command 和 Automation。
- Outbox 消费者离线或崩溃时租约最终过期，事件保留在本地并可重领；重试达到上限后进入可诊断死信，不形成无限热循环。

## 11. 当前本地备份与恢复

- 桌面进程每 15 分钟检查一次调度；距最新备份达到 6 小时才创建新备份，默认最多保留 12 份。
- 备份使用 SQLite Online Backup API 生成包含 WAL 已提交状态的一致副本；先写入 `.partial`，通过当前 Schema 与 `PRAGMA integrity_check` 后同步并原子发布。
- 创建失败会清理临时副本，不覆盖现有数据库或已发布备份；后台调度错误保留在进程健康状态并显示在设置页。
- 用户只能从宿主列出的有界备份 ID 中选择恢复。请求在写盘前重新验证备份，并以耐久标记安排到下次启动。
- 恢复必须发生在任何 SQLite 连接打开之前：验证源备份、复制到 staged、再次验证、保留 rollback 后原子激活；失败时恢复原数据库。
- 保留策略不得删除挂起恢复所引用的备份；成功激活后清理 rollback、WAL/SHM sidecar 和恢复标记。
- 数据库验证或挂起恢复失败时，应用仍启动 Control Center，但所有运行时、Profile、权限和 Outbox 改用内存隔离仓库；自动备份和正常写操作暂停，原数据库保持原样。
- 隔离恢复模式只允许读取备份健康状态，并从宿主列出的已验证备份安排下次启动恢复；前端仅收到稳定原因码，不暴露绝对路径或原始 SQLite 错误。
- 当前范围仅为单设备本地备份，不等同于数据库修复工具、用户数据导出、跨设备复制或云备份；已提供不含数据库正文和路径的脱敏诊断摘要包，人工数据提取仍需实现。
