# Nimora 功能演进与创新储备

> 版本：0.1.0-draft  
> 更新日期：2026-07-17

## 1. 目的

本文件记录完整产品范围中值得持续建设的能力，防止架构封闭，也防止施工顺序被误解为范围删减。任何能力进入具体里程碑前，必须按 [`FUTURE_EVOLUTION_GOVERNANCE.md`](FUTURE_EVOLUTION_GOVERNANCE.md) 完成价值、隐私、离线、性能、UI、兼容和退出策略评审。

## 2. 优先级定义

- **Foundation**：现在必须预留契约或数据边界。
- **Expansion**：核心稳定后可作为官方模块或扩展交付。
- **Exploration**：需要验证价值、成本或伦理风险。

## 3. 功能地图

| 方向 | 能力 | 优先级 | 设计边界 |
|---|---|---|---|
| 多宠系统 | 多角色编队、主从关系、共享/独立状态、空间避让 | Foundation | 每宠独立身份与资源预算，事件必须带 Pet ID |
| 场景系统 | 工作、休息、直播、游戏、夜间场景编排 | Foundation | Scene 是可组合 Profile，不复制全量配置 |
| 跨设备 | 手机遥控、局域网面板、状态同步、远程通知 | Expansion | 桌面端离线独立，配对、E2E 加密、最小 Scope |
| 创作者直播 | OBS 透明输出、虚拟摄像头、OSC、Stream Deck、弹幕 | Expansion | 直播输出与桌面浮层分离，弹幕视为不可信输入 |
| 智能家居 | Home Assistant、MQTT、Matter Bridge、场景联动 | Expansion | 设备写操作高风险，预览、限频、撤销或补偿 |
| 开发者工作流 | IDE、Git、CI、终端、Issue、代码评审反馈 | Expansion | 只读优先，工作区路径与命令执行细粒度授权 |
| 角色共创 | 动作/表情适配补丁、换装组合、协作项目 | Expansion | 不修改原包，保留来源、许可证和合并历史 |
| 社区生态 | 收藏、评价、兼容报告、精选清单、举报和撤回 | Expansion | 不以付费或热度降低安全审核标准 |
| 数字商品 | 免费、买断、订阅、打赏、组织授权、离线宽限 | Exploration | 用户数据与许可证分离，支持迁移和退款后安全降级 |
| 辅助陪伴 | 工作节律、情绪记录、习惯和低压力提醒 | Expansion | 不做医疗诊断，不制造依赖或羞辱式激励 |
| 家庭模式 | 多账户、访客、儿童保护、安静时段 | Exploration | 默认最小数据、家长控制透明、禁止隐蔽监控 |
| 空间桌面 | 多屏区域、窗口锚点、应用场景、虚拟桌面 | Expansion | 不持续截屏；窗口感知通过最小平台元数据 |
| 语音交互 | 本地唤醒、按键说话、TTS、字幕和声线包 | Expansion | 麦克风持续指示、本地优先、音频保留可配置 |
| 动捕面捕 | 摄像头、ARKit 类数据、OSC/VMC 输入、录制回放 | Expansion | 原始影像默认不保存，追踪数据独立授权 |
| 内容生成 | AI 台词、动作建议、贴图辅助、自动绑定 | Exploration | 生成内容有来源、可编辑、版权提示和人工发布确认 |
| 无障碍角色 | 字幕宠物、视觉提醒、低动作角色、单手操作 | Foundation | 无障碍是等价体验，不是简化残缺版本 |

## 4. 必须立即预留的架构能力

- 所有宠物状态、Command、Event、资源实例和日志支持明确 `petId`，但单宠 UI 可隐藏。
- Profile 可组合为 Scene，支持临时覆盖、优先级、自动进入与退出恢复。
- Gateway Client Identity 可代表本机工具、移动设备和组织服务，权限不能共享一个万能 Token。
- 输出目标抽象为 Desktop Overlay、Capture Surface、Virtual Camera 等 Sink，避免直播功能侵入核心窗口。
- 外部输入带来源、信任等级、速率与内容分类，弹幕、Webhook、AI 和设备消息不能直接执行 Command。
- 角色项目支持派生关系与适配补丁，为换装、协作和模型升级保留非破坏性合并能力。

## 5. 新的产品设计机会

### 5.1 场景导演

用户可以把角色、布局、Profile、自动化、音乐、灯光和输出目标组合成 Scene；一键进入“专注”“直播”“睡前”“编码”等状态。切换前预览系统影响，退出时恢复原状态。

### 5.2 宠物能力卡

把复杂 Command、脚本和自动化包装成可理解的能力卡：展示用途、输入、权限、来源、最近运行和快捷操作。普通用户用卡片组合，高级用户展开代码与 Schema。

### 5.3 角色健康检查

为第三方模型生成统一报告：动作完整度、表情覆盖、许可证、性能、可访问性、兼容平台和安全扫描。报告可被用户理解，也可作为 Registry 的可验证元数据。

### 5.4 安静智能

建立全局注意力预算，将气泡、通知、声音、动作和 Agent 主动建议统一排队。依据全屏、会议、输入活跃、勿扰和用户反馈降级，避免多个模块各自打扰。

### 5.5 可回放桌面自动化

用脱敏事件记录、虚拟时间和 Command dry-run 复现脚本及规则问题。用户可分享最小诊断场景而不泄露真实文件、窗口标题和消息内容。

## 6. 不建议纳入核心的能力

- 无限制运行任意系统脚本：保留为用户显式配置的外部工具桥接，不进入默认 Host。
- 自建封闭模型格式：优先 VRM/glTF、Live2D Adapter 和公开中间语义。
- 平台级社交聊天：除非验证独立价值，否则通过 Connector 承载，避免扩大审核与未成年人风险。
- 强制云账户和云同步：不得破坏本地优先、离线使用和用户迁移权。
- 以“情感陪伴”为由诱导付费、连续签到或限制删除记忆。

## 7. 进入路线图的门槛

每项候选能力必须回答：

1. 对哪类用户解决什么高频问题？
2. 为什么应由核心提供，而不是扩展？
3. 无网络、无账户、无 AI 时如何工作或降级？
4. 最敏感的数据和最危险的副作用是什么？
5. 对启动、内存、GPU、电池和安装包的预算是多少？
6. Windows/macOS 的差异与不可用能力如何表达？
7. 如何测试、观测、回滚、弃用和迁移？
8. UI 的空态、失败、权限、恢复和无障碍状态是否完整？
