# DeskPet 安全与隐私规范

> 版本：0.1.0-draft  
> 更新日期：2026-07-17  
> 本文中的“必须”是发布阻断条件。

## 1. 威胁模型

主要威胁包括恶意扩展、被接管的 Registry、供应链包、SSRF、Token 泄露、Prompt Injection、事件风暴、路径穿越、权限诱导和审计中泄露敏感数据。

## 2. 安全边界

- Core、Capability Broker 和 Secure Store 属于可信计算基。
- WebView、扩展、外部资源、Agent 输出和外部事件均不可信。
- 进程隔离不等于权限隔离。
- 所有跨边界输入必须经过 Schema、大小、速率和语义验证。

## 3. 权限原则

- 默认拒绝、最小权限、按目标授权、可撤销、可审计。
- 安装授权与运行时确认分离。
- Danger 操作不能被“始终允许”绕过，除非组织策略明确授权。
- 权限扩大视为需要重新授权的更新。
- Agent 不能获得高于发起用户和扩展本身的有效权限。

## 4. 网络安全

- Gateway 默认关闭并监听 `127.0.0.1`/`::1`。
- 局域网监听需要明确开启、配对和可见警告。
- 出站远程目标必须在用户确认的 destination policy 中。
- HTTP 必须限制重定向、DNS 重绑定、私有网段访问和响应大小。
- URI Scheme 只能打开预览或确认页面，不能静默安装或执行命令。
- Connector 必须限制速率、并发、队列、超时和最大载荷。

## 5. 密钥与身份

- API Key、Token 和签名私钥只能保存于 OS Secure Store 或外部密钥服务。
- 日志、配置导出、崩溃报告和事件不得包含真实密钥。
- Gateway Token 支持 Scope、过期、撤销、客户端身份和最近使用信息。
- 包身份由 `publisher + package id + signing key` 共同确定。

## 6. 扩展隔离

- 禁止扩展直接使用 Node 文件、网络、进程和原生模块。
- 敏感能力通过 Capability Broker RPC 调用。
- 每个扩展拥有独立存储命名空间、CPU、内存、定时器和消息配额。
- 原生扩展进入独立高风险进程，并使用平台级沙箱能力。
- 用户本地脚本与下载脚本同样属于不可信代码；仅通过注入 SDK 调用 Command 与 Capability Broker，不因来源为本地而放宽权限。
- 模型与资源导入器在受限进程中处理不可信文件，限制解包、解析、转换、外部 URI、CPU、内存、时间和输出大小。
- 连续崩溃、超额或异常网络行为触发 quarantine。

## 7. 数据分类

| 等级 | 示例 | 默认策略 |
|---|---|---|
| Public | 包名、公开事件类型 | 可显示，不自动出站 |
| Internal | Profile、技能状态 | 需明确订阅 |
| Sensitive | 坐标、应用名称、习惯 | 默认脱敏，逐项授权 |
| Restricted | 对话原文、文件内容、Token | 默认禁止出站 |

每个 Connector 必须展示发送目标、事件类型、字段分类、频率和最近一次发送结果。

## 8. AI 安全

- 外部文本、网页和 Source Connector 数据被视为不可信内容，不能成为系统指令。
- 工具选择后重新计算有效风险，不相信模型给出的风险标签。
- Danger 工具逐步确认，并展示实际参数。
- Agent 任务设置最大步骤、时间、Token、工具次数和费用预算。
- 记忆写入需要策略过滤；敏感推断应要求用户确认。
- Provider 请求前展示或记录将发送的数据分类。

## 9. 审计

审计记录主体、动作、目标、结果、时间、Trace ID 和数据分类，但不记录密钥和默认受限正文。审计文件需轮转、完整性保护，并允许用户导出和清除。

## 10. 安全模式

安全模式必须：

1. 停止 Gateway。
2. 断开 Source、Sink 和 Duplex Connector。
3. 停止第三方代码和 Agent Worker。
4. 保留官方默认资源、Pet Runtime 和本地设置入口。
5. 展示进入原因和逐项恢复选项。
