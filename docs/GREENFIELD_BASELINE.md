# Nimora 全新项目基线

> 版本：0.1.0-draft  
> 更新日期：2026-07-17  
> 状态：首个稳定版前强制执行

## 1. 项目事实

Nimora 是全新项目，不存在已发布旧版本、旧用户数据、旧插件生态或需要继续支持的历史协议。开发期文件、数据库、Manifest 和测试夹具都不是公开兼容承诺。

## 2. 首版开发规则

- 每类数据、API、事件、Manifest 和 SDK 只保留一个现行契约。
- 契约调整时直接同步修改实现、测试夹具和文档，不保留旧字段、旧分支或双写逻辑。
- 开发数据库结构变化时直接重建当前 Schema；不得为未发布结构编写迁移链。
- 不添加 `legacy`、`deprecated`、兼容别名、旧品牌重定向或猜测性版本转换。
- 不因未来可能变化而预建 Adapter；只有第二个真实实现出现时才提炼公共抽象。
- 版本字段用于识别和拒绝未知输入，不证明存在历史版本，也不要求迁移实现。
- 本地目录名、早期草稿名和实验性提交不构成产品兼容面；公开品牌统一为 Nimora。

## 3. 当前版本治理

首个稳定版发布前，破坏性变更的正确处理方式是：

```text
修改唯一契约
→ 更新生产实现
→ 更新测试夹具
→ 重建开发数据
→ 更新文档与示例
```

不得采用：

```text
保留旧契约
→ 增加兼容分支
→ 编写开发期迁移
→ 长期维护两套行为
```

## 4. 稳定版后的触发条件

只有首个稳定版真实发布并产生用户数据或第三方生态后，才根据实际发布物设计升级、迁移、弃用和回滚策略。届时必须以真实版本、真实数据样本和真实扩展契约为输入，不得复活开发期中间结构。

## 5. 场景与能力

`companion`、`work`、`focus`、`creator`、`developer`、`presentation` 和 `offline` 是体验预设，不是产品版本或兼容层。它们可以调整布局、通知、动效和资源预算，但不得永久删减用户已经安装并授权的能力；能力控制只由 Capability Gateway、用户授权和全局安全模式决定。

## 6. 评审门禁

任何包含下列内容的变更都必须说明对应的真实已发布对象，否则拒绝合并：

- 数据迁移步骤或版本转换器。
- 兼容旧字段、旧命令、旧包格式或旧品牌的分支。
- `deprecated` API 或弃用期限。
- 为假想旧客户端保留的协议降级。
- 为尚不存在的第二实现建立的复杂抽象层。
