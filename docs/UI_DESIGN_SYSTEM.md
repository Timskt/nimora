# Nimora UI 设计系统与质量门禁

> 版本：0.1.0-draft  
> 更新日期：2026-07-17  
> 状态：强制设计与实现基线

## 1. 目标

Nimora 的 UI 必须同时做到精致、克制、有生命感、可信任和长期可维护。美观不是发布后的润色项：页面在视觉层级、组件状态、动效、文案、无障碍或跨平台一致性上不达标时，功能不得进入 stable。

本规范约束桌宠浮层、控制中心、Creator Studio、权限界面、首次引导、调试工具和扩展贡献 UI。角色包可以改变角色与主题表达，但不能破坏系统信息层级、安全语义和可恢复入口。

## 2. 设计原则

1. **角色是情感焦点，UI 是安静舞台**：控制界面不与宠物争夺注意力。
2. **细节形成品质**：对齐、间距、字体、圆角、阴影和状态变化必须来自令牌。
3. **每个状态都被设计**：默认、Hover、Pressed、Focus、Selected、Loading、Disabled、Success、Warning、Error、Offline 和 Empty 不得临时拼凑。
4. **高级但不晦涩**：普通用户看到结论和下一步，开发者可逐层展开参数、事件和 Trace。
5. **动效解释关系**：动画表达来源、去向、层级和状态，不作为无意义装饰。
6. **信任高于主题**：权限、危险操作、隐私和系统错误不能被皮肤伪装。
7. **跨平台一致而不僵化**：保留 Nimora 品牌，同时尊重 Windows/macOS 的窗口、菜单、输入和字体习惯。

## 3. 设计令牌

所有视觉值必须来自语义 Token，禁止业务组件散落任意颜色、字号、阴影和层级数字。

### 3.1 Token 层级

```text
Primitive Token → Semantic Token → Component Token → Theme Override
```

- Primitive：色阶、尺寸、字重、透明度和基础曲线，不直接供页面使用。
- Semantic：`surface.canvas`、`text.primary`、`border.focus`、`status.danger` 等意图。
- Component：`button.primary.background.hover` 等组件特定值。
- Theme Override：仅覆盖允许主题化的 Token；安全色、焦点可见性和最低对比度不可降低。

### 3.2 基础尺度

- 间距采用 4 px 基线，常用值为 4、8、12、16、20、24、32、40、48、64。
- 页面栅格为 8 px；图标与文字允许 4 px 微调，不允许肉眼随意偏移。
- 圆角建议层级：小控件 6–8 px、卡片 10–12 px、浮层 12–16 px、胶囊使用完整半径。
- 边框默认 1 px；高 DPI 下保持物理清晰，不出现半像素模糊。
- 阴影只表达悬浮层级，不用于弥补低对比度边界。

### 3.3 色彩

- 每个主题必须定义 canvas、surface、elevated、overlay、primary、secondary、text、border 和 status 色组。
- 正文、次要文字、禁用状态、图标和焦点环分别验收对比度。
- 状态不能只依赖颜色，必须结合图标、形状或文字。
- 大面积高饱和色只用于角色舞台、短暂庆祝或明确主操作。
- 深色模式不是颜色反相；重新评估阴影、边界、图片光晕和角色轮廓。

### 3.4 字体

- UI 使用平台优先的可读无衬线字体；代码、事件 ID、端口和数值使用等宽字体。
- 最小常规正文不低于 13 px；主要正文建议 14–16 px。
- 层级通过字号、字重、颜色和间距共同建立，单页可见字重不超过四档。
- 中英文、数字和标点基线必须经过真实内容截图检查。
- 字号缩放 200% 时不得截断关键操作或用缩小字号补救。

## 4. 布局与层级

### 4.1 控制中心

- 采用稳定的应用框架：导航、页面标题、主内容、上下文操作和全局状态位置不随页面漂移。
- 页面主操作最多一个；次要操作降低视觉权重，危险操作与常规主操作分离。
- 内容最大宽度按任务决定，设置页避免超长行，数据页允许高密度全宽布局。
- 相关字段组成清晰分组；分组间距大于组内间距。
- 高级参数渐进展开，但搜索必须能命中隐藏设置并显示所在路径。

### 4.2 桌宠浮层

- 宠物、气泡、快捷菜单和拖拽反馈是不同层级，不互相遮挡关键内容。
- 气泡自动避让屏幕边缘、任务栏/Dock、当前输入焦点和主要窗口区域。
- 点击穿透启用后，托盘和全局热键始终可恢复交互。
- 宠物缩放、阴影和轮廓必须适配浅色、深色、复杂壁纸和多显示器 DPI。
- 多宠时使用空间预算和队列，禁止气泡互相覆盖或所有角色同时抢焦点。

### 4.3 Creator Studio

- 画布是第一视觉焦点，时间轴、层级树、检查器和问题面板形成稳定工作区。
- 面板支持停靠、折叠、调整和恢复默认，布局按项目与用户分别保存。
- 选择态在画布、层级树、时间轴和属性面板同步。
- 修改、未保存、验证中、错误和只读状态在标题与相关对象上同时可见。
- 高密度界面仍保持 32×32 px 最小精确点击目标；常规界面目标不低于 40×40 px。

## 5. 核心组件规范

每个组件必须有 Anatomy、Token、尺寸、内容规则、状态矩阵、键盘行为、ARIA 语义、加载策略、错误策略和测试夹具。

必须建立并复用：

- Button、Icon Button、Split Button、Link。
- Text Field、Text Area、Search、Number、Select、Combobox、Slider、Color、Keybinding。
- Checkbox、Radio、Switch、Segmented Control。
- Card、List、Tree、Table、Virtual List、Tabs、Accordion。
- Tooltip、Popover、Menu、Context Menu、Dialog、Drawer。
- Toast、Banner、Inline Message、Progress、Skeleton、Empty State。
- Command Palette、Permission Prompt、Risk Preview、Audit Row。
- Pet Bubble、Action Chip、Task Progress、Offline Badge。
- Code Editor、Schema Form、Timeline、Node Graph、Inspector。

### 5.1 状态细节

- Hover 只作为辅助反馈，核心能力不能依赖 Hover 才可发现。
- Pressed 必须有即时视觉响应，不等待异步结果。
- Focus ring 不得被 `outline: none` 移除；与 Selected 状态可同时辨识。
- Loading 保持组件尺寸稳定，防止布局跳动；超过 300 ms 才显示加载反馈。
- Disabled 必须解释原因；可修复情况优先使用可点击但会解释的受限状态。
- Error 紧邻问题字段并保留用户输入；页面顶部提供错误摘要和定位。
- Destructive 操作只在后果不可逆或高影响时使用危险样式，避免红色滥用。

## 6. 关键流程标准

### 6.1 首次启动

- 60 秒内完成选择角色、首次互动和基础偏好。
- 不在首次启动批量索取通知、网络、麦克风等非即时必要权限。
- 提供跳过、返回、稍后设置和离线继续。
- 默认角色加载失败时仍能完成控制中心与修复流程。

### 6.2 模型与扩展导入

- 固定为“来源 → 扫描 → 兼容与权限 → 预览 → 安装结果”步骤。
- 许可证、性能、安全、缺失动作分别展示，不合并成模糊“存在问题”。
- 普通用户看到建议与自动修复；创作者可展开原始报告。
- 安装失败不清空选择和映射；提供重试、导出诊断和安全退出。

### 6.3 权限与危险操作

- 明确显示请求者、动作、数据、目标、期限、后果和撤销入口。
- 允许一次、本次会话、始终允许必须视觉区分且不默认选择最高权限。
- 高风险批量操作展示影响数量、不可逆部分和预览差异。
- 输入确认只用于真正高危且不可撤销操作，不能用来掩盖糟糕的信息设计。

### 6.4 离线与故障

- 顶层离线提示不遮挡内容，说明哪些功能仍可用。
- 局部失败局部展示，不把整页替换成错误页。
- 自动降级必须告知当前模式、影响和恢复方式。
- 崩溃恢复界面优先保留数据、草稿和最近操作上下文。

## 7. 动效与微交互

- 交互反馈 80–120 ms，面板 160–220 ms，页面过渡 180–260 ms。
- 使用统一 motion token：duration、easing、distance、stagger 和 reduced-motion fallback。
- 列表插入、删除与排序必须让用户看懂对象去向。
- 宠物动作遵循准备—动作—恢复；系统 UI 不模仿角色的大幅弹性动效。
- 同时竞争注意力的循环动画不超过一个；所有循环动画可停止。
- 减少动画模式取消位移、缩放、视差、抖动和闪烁，保留必要淡入与状态文字。

## 8. 图标、插画与声音

- 图标采用统一视框、描边、端点和光学尺寸；禁止混用不同视觉体系。
- 无文字 Icon Button 必须有 Tooltip 与可访问名称。
- 空状态插画服务于解释与行动，不占据超过主要任务的视觉权重。
- 角色艺术与系统图标分层维护，角色包不能替换安全警告图标。
- 声音与视觉反馈同步，不可成为唯一反馈；相同提示有冷却和独立音量控制。

## 9. 响应式与跨平台

- 验收宽度至少覆盖 720、960、1280、1440、1920 px 和 200% 缩放。
- 不以隐藏关键功能作为窄屏适配；使用折叠、抽屉、重新排序和分步流程。
- Windows/macOS 分别验收窗口控制区、安全区域、滚动条、快捷键、菜单和字体渲染。
- 多显示器切换不改变物理大小，不把对话框留在已断开的屏幕。
- 触控板、鼠标、键盘和高精度笔输入均不得出现 Hover 锁死或拖拽丢失。

## 10. 扩展 UI 治理

- 扩展只能使用受控 Contribution Slot 和官方组件，不能注入全局 CSS、脚本或覆盖系统 DOM。
- 扩展 Theme 只获得允许 Token；不能降低对比度、隐藏来源、改变危险语义或仿冒系统对话框。
- 设置面板必须声明标题、说明、关键字、权限关联、空/错/加载状态和卸载行为。
- 扩展 UI 崩溃时显示隔离错误卡片，不终止页面；用户可停用扩展并导出诊断。

## 11. 设计到代码流程

1. 用户流与状态清单评审。
2. 低保真信息架构与交互评审。
3. 使用真实文案、极端数据和多语言的高保真设计。
4. Design Token 与组件映射评审。
5. Storybook/组件实验室实现全部状态。
6. 键盘、读屏、缩放、主题和减少动画检查。
7. Windows/macOS 实机视觉检查。
8. 截图基线、交互测试和设计签字。

设计稿不得只展示理想成功态。每个功能必须随设计提交状态矩阵、空态、失败、离线、权限拒绝、长文本、极值和恢复路径。

## 12. 自动化质量门禁

- Token lint：禁止未登记颜色、字号、z-index、阴影和间距。
- Component lint：禁止业务页面复制基础控件或绕过统一组件。
- Accessibility：axe 类规则、键盘路径、焦点顺序、名称与对比度。
- Visual regression：浅色、深色、高对比度、200% 缩放和中英文截图。
- Motion regression：减少动画模式、加载稳定性和布局位移。
- Cross-platform screenshot：Windows/macOS 关键页面和桌宠边界。
- Performance：控制中心交互响应、长列表、时间轴和模型预览帧率预算。

视觉差异必须由设计负责人批准更新基线，禁止为让 CI 通过而批量接受截图。

## 13. 人工设计验收评分

每个关键流程按 0–2 分评分：

| 维度 | 0 分 | 1 分 | 2 分 |
|---|---|---|---|
| 层级 | 无焦点、混乱 | 基本清晰 | 主次准确且自然 |
| 对齐与节奏 | 明显散乱 | 少量瑕疵 | 一致、精确、舒适 |
| 状态完整 | 缺关键状态 | 基本覆盖 | 全状态且恢复明确 |
| 文案 | 含糊或技术化 | 可理解 | 简洁、可信、可行动 |
| 动效 | 干扰或跳变 | 基本顺畅 | 克制且解释关系 |
| 无障碍 | 无法完成 | 可完成 | 等价、清晰、舒适 |
| 跨平台 | 明显异常 | 可用 | 符合平台且品牌一致 |
| 情感品质 | 生硬或廉价 | 中性 | 有生命感但不打扰 |

任一维度为 0 或总分低于 13/16 不得进入候选发布。权限、恢复路径、键盘操作和对比度属于一票否决项。

## 14. 禁止项

- 禁止随意渐变、玻璃拟态、霓虹光、过量阴影和无语义装饰。
- 禁止用 Emoji 代替核心产品图标。
- 禁止没有 Loading、Empty、Error 和 Offline 状态的页面。
- 禁止仅凭颜色表达成功、警告、错误和选择。
- 禁止多套相似组件、局部重置字体或业务组件自定义 z-index 战争。
- 禁止不可撤销的点击即执行、隐藏退出路径和默认勾选高权限。
- 禁止把开发调试 UI 直接暴露给普通用户，也禁止高级模式牺牲视觉质量。

## 15. 发布验收清单

- 全部界面只使用批准 Token 与组件。
- 普通、角色、极客、创作者、开发者模式的信息密度均合理。
- 浅色、深色、高对比度和自定义主题不破坏状态语义。
- 中英文、长文本、空数据、极端数据和错误数据布局稳定。
- 键盘、读屏、200% 缩放、减少动画完整可用。
- 桌宠在复杂壁纸、多屏、DPI、全屏和点击穿透下可见且可恢复。
- Creator Studio 在长时间编辑、大模型和多面板下保持流畅。
- 关键流程通过视觉回归和人工评分，没有未经批准的像素偏差。

### Agent 工作台验收补充

- 对话任务区与运行检查器在宽屏为主次双栏，900px 以下转为单栏，720px 以下操作区纵向排列。
- Provider 的本地性、联网需求、凭据与费用必须在首屏可见；Tool ID 长文本允许换行，不造成横向滚动。
- 只读使用薄荷语义，需确认写操作使用暖琥珀语义，同时保留 `R/W` 与文字标签，禁止只靠颜色表达。
- 完成结果同时展示内容、Task 状态、Token 和费用；失败不得保留虚假的成功状态，写操作不得出现“自动执行”暗示。
- 写工具确认卡必须展示 Tool ID、格式化实际参数、风险、单次有效与过期时间，并提供同等清晰的拒绝入口；Approval 指纹等内部授权材料不得进入 DOM。
- 多工具确认组必须维持 Provider 原始顺序；部分批准后布局稳定收敛，最终态恢复简洁对话结构，拒绝态明确说明整组零副作用。
- 浏览器预览 Scripted Provider 只用于视觉验证；截图必须能识别当前 Provider，禁止用预览结果冒充生产 Ollama 验收。
