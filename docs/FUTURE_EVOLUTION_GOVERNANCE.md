# Nimora 用户需求与技术演进治理

> 版本：0.1.0-draft  
> 更新日期：2026-07-17  
> 所有者：产品、架构、生态与可靠性负责人

## 1. 目标与原则

本规范保证 Nimora 面向广大用户后，新增需求、新设备、新模型格式、新 AI 能力和新运行时可以持续进入产品，而不破坏离线能力、安全边界、数据所有权、UI 一致性和既有生态。

- 有明确用户价值的能力必须进入需求地图；优先级只决定施工顺序，不等于删除范围。
- Core 依赖稳定的领域语义，不依赖某个 AI 厂商、Renderer、模型格式、操作系统 API 或 JavaScript 运行时。
- 新技术先证明可替换、可降级、可观测和可退出，再成为默认实现。
- 云端增强不得成为桌宠、资源、规则、用户代码和本地数据的强制前置条件。
- 新能力必须同时设计普通模式、专家模式、无障碍路径、离线路径和失败路径。
- 首个稳定版前维护唯一当前契约；真实发布后才依据实际用户数据和生态包制定迁移与弃用周期。

## 2. 持续需求雷达

### 2.1 用户群与高概率新增需求

| 用户群 | 高概率需求 | 必须预留的产品边界 |
|---|---|---|
| 普通用户 | 更简单的引导、低打扰、跨设备、备份恢复、主题与角色市场 | 渐进披露、默认离线、本地导入导出、可撤销配置 |
| 二次元与角色爱好者 | Live2D、VRM、换装、表情、语音、动作、陪伴记忆、多宠互动 | Character/Renderer/Voice/Behavior 分离，派生关系和许可证链 |
| 创作者与主播 | 动捕、面捕、OBS、透明输出、虚拟摄像头、弹幕、商店发布 | Capture Sink、Tracking Source、审核报告、版权与性能预算 |
| 极客与自动化用户 | 本地协议、Home Assistant、MQTT、快捷键、Webhook、硬件设备 | Connector Adapter、Capability Gateway、速率限制和 dry-run |
| 开发者 | SDK、CLI、类型生成、调试器、事件回放、IDE 集成、私有 Registry | 版本化 Schema、稳定 Command/Event、模拟器和兼容测试套件 |
| 团队与组织 | 策略、批量部署、审计、代理配置、私有资产与扩展分发 | Policy Provider、签名信任根、无秘密日志和可导出审计 |
| 无障碍用户 | 字幕、读屏、键盘、低动作、高对比、认知负担控制 | 等价语义操作、设计令牌、注意力预算和自动化可达性门禁 |
| 隐私敏感用户 | 完全离线、无遥测、数据可见、可删除、可迁移 | Local-first、显式网络指示、数据目录和一键导出/擦除 |

### 2.2 需求来源

需求雷达持续吸收：应用内反馈、崩溃与性能诊断、无障碍审计、社区讨论、Registry 搜索无结果、扩展作者提案、支持工单、平台 API 变化、安全通告和竞品能力。遥测必须自愿、最小化、可关闭，不得用隐蔽行为分析替代用户访谈。

每条候选需求记录：目标用户、要完成的任务、当前阻碍、频率、严重度、离线预期、数据敏感度、平台范围、可访问性影响、生态价值、证据来源和成功指标。重复请求合并到同一 Problem ID，避免按声音大小而非问题价值决策。

### 2.3 从反馈到交付

```text
反馈与证据 → Problem Brief → 需求地图 → RFC/ADR → 契约原型
            → 垂直切片 → 真实 UI/性能/故障验证 → 稳定能力
```

- Problem Brief 描述用户问题，不预设技术方案。
- 涉及公开扩展点、持久数据、安全边界或进程模型时必须有 RFC/ADR。
- 第一个实现可直接落在清晰模块内；第二个真实实现出现后才提炼公共抽象。
- 垂直切片必须包含 UI、Core、持久化、权限、日志、离线、恢复、测试和文档，不接受只有演示页面的“支持”。
- 尚未施工的高价值能力保留在 `FEATURE_EVOLUTION.md` 与 `IMPLEMENTATION_STATUS.md`，不得用“未来版本”从范围中移除。

## 3. 面向变化的稳定架构接缝

| 变化方向 | 稳定语义/Port | 可替换 Adapter |
|---|---|---|
| 角色渲染 | Character、Action、Expression、Gaze、LipSync | Sprite、Live2D、VRM、glTF、WebGPU、原生 Renderer |
| AI 与 Agent | Model Provider、Tool、Context、Policy、Trace | 本地模型、云模型、系统模型、未来推理运行时 |
| 用户代码 | Worker Protocol、Capability Gateway、Execution Budget | JavaScript 沙箱、WASM Component、受控 DSL |
| 自动化 | Trigger、Condition、Command、Event、Schedule | 本地规则、系统快捷指令、外部编排器 |
| 外部系统 | Source/Sink Connector、Identity、Scope | HTTP、MCP、MQTT、Matter、IDE、日历、消息平台 |
| 输出目标 | Render/Audio Sink | 桌面浮层、OBS、虚拟摄像头、移动端、空间计算设备 |
| 存储与同步 | Repository、Outbox、Backup、Sync Provider | SQLite、本地文件、局域网同步、端到端加密云同步 |
| UI 能力 | Contribution、Design Token、View Model | React WebView、原生面板、移动控制端、可访问性替代视图 |

模块只能通过 Manifest 声明贡献，通过 Capability Gateway 调用其它模块公开能力。禁止跨模块读写内部数据库、文件或进程；这样未来替换框架或运行时不会把生态绑定到实现细节。

## 4. 技术雷达

技术雷达每季度和重大平台发布后更新，使用四个状态：**观察**、**试验**、**采用**、**退出**。状态是证据结论，不是流行度排名。

重点持续观察：

- WebGPU、原生 GPU 合成、增量场景图和低功耗动画调度。
- VRM、glTF 扩展、Live2D SDK、开放动作/表情与动捕协议。
- WASM Component Model、能力安全运行时、确定性脚本和可移植插件 ABI。
- 本地小模型、多模态模型、流式语音、设备 NPU 与系统级模型 API。
- MCP 及后续工具协议、Agent-to-Agent 协议、可验证工具调用与策略证明。
- CRDT、Local-first 同步、端到端加密协作和离线许可证证明。
- Windows、macOS、Linux 的窗口、托盘、无障碍、GPU、签名和沙箱变化。
- 空间计算、AR 桌面、移动伴侣、可穿戴设备和新型输入输出终端。

每个雷达条目必须记录：解决的问题、替代对象、成熟度、许可证、维护者健康、平台覆盖、包体/内存/GPU/能耗、供应链风险、隐私影响、可访问性、退出成本、实验负责人和复审日期。

## 5. 新技术准入流水线

1. **隔离实验**：在 `experiments/` 或独立分支验证，不进入生产依赖图，不读取真实用户数据。
2. **契约适配**：证明可通过现有 Port 接入；若不成立，先用 ADR 判断是领域缺失还是技术泄漏。
3. **对照基准**：与当前实现比较启动时间、帧时间、内存、GPU、能耗、包体、崩溃和离线行为。
4. **威胁与供应链审查**：检查权限、沙箱、网络、签名、SBOM、许可证、维护风险和故障半径。
5. **真实体验验证**：在目标平台完成截图、交互、键盘、读屏、减少动画、高对比和弱设备验证。
6. **双实现试运行**：通过内部 Feature Flag 或 Adapter 选择器灰度；安全策略不可被 Flag 关闭。
7. **退出演练**：验证禁用新实现后数据仍可读、核心仍可离线运行、用户可回退且无资产锁定。
8. **采用或退出**：证据达到预算才设为默认；不达标则移除实验代码、依赖、Flag 和文档承诺。

## 6. 兼容与生态演进

### 6.1 稳定版前

- 直接修正唯一当前 Schema、SDK、夹具和文档。
- 不创建虚构迁移链、旧品牌桥接、废弃 API 或兼容层。
- 实验包必须明确不可发布到公共 Registry。

### 6.2 真实稳定版后

- 公开契约按能力独立版本化，宿主声明支持矩阵，包声明最低/最高兼容范围。
- 破坏性变更必须提供自动检查器、迁移工具、双版本测试夹具、遥测无关的生态影响报告和明确停止支持日期。
- Registry 在安装前展示兼容、权限、性能和签名结论；撤回版本保留安全说明但停止新安装。
- SDK 至少维护当前稳定线和一个有真实生态使用的前一稳定线，具体周期由发布 ADR 决定，不提前虚构年限。
- 数据导出格式保持可读和有 Schema；更换同步、AI 或 Renderer Provider 不得锁住用户资产与历史。

## 7. 防止架构僵化的强制检查

每个重大设计必须回答：

1. 如果供应商、框架或格式明天停止维护，哪个 Adapter 可被替换？
2. 完全断网时哪些能力仍工作，用户会看到什么降级？
3. 新模块如何通过 Manifest 和 Gateway 复用已有能力，而不访问内部实现？
4. 普通用户是否无需理解技术细节，高级用户是否仍能检查和控制？
5. 数据能否导出、删除、恢复并被下一实现读取？
6. 失败会否拖垮 Core；是否有超时、取消、背压、熔断、隔离和安全模式？
7. UI 在不同屏幕、语言、缩放、输入方式和减少动画设置下是否仍完整？
8. 如何用契约测试证明第三方实现，而不是依赖人工口头兼容？

任一问题无答案，不得将能力标记为完整实现。

## 8. 验证与指标

- 需求质量：Problem ID 到发布证据可追踪，重复需求合并率，已接受需求的完成周期。
- 生态健康：活跃包兼容率、安装失败率、权限拒绝后的可恢复率、扩展崩溃隔离率。
- 技术可替换性：Adapter 契约套件通过率、Provider 切换成功率、退出演练恢复时间。
- 离线与数据：冷启动离线成功率、备份恢复成功率、导出可读率、网络禁用时核心门禁。
- 体验质量：关键路径任务成功率、可访问性阻断数、帧时间、低端设备资源预算和视觉回归差异。

指标只用于发现系统问题，不得把陪伴时长、连续签到或情绪依赖设计为操纵性增长目标。

## 9. 文档与治理产物

- `FEATURE_EVOLUTION.md`：完整功能机会地图。
- `IMPLEMENTATION_STATUS.md`：真实实现、证据和缺口。
- `adr/`：已接受的架构与安全决定。
- `rfcs/`：进入评审的公开能力提案；首个真实 RFC 出现时创建目录。
- `TECHNOLOGY_RADAR.md`：首批真实评估发生时生成，不预填虚构结论。
- 契约兼容报告、基准结果和退出演练记录必须随对应实现提交。

本规范本身不能代替实现。每次迭代都应从状态矩阵选择一个可闭环纵切，同时保留其余有价值需求，持续把未来适应能力转化为已验证代码。
