# ADR-008：宠物运行状态采用版本化 SQLite 快照

> 状态：Accepted
> 日期：2026-07-17

## 背景

Nimora 必须在完全离线时保存宠物状态，并在应用重启后可靠恢复。领域核心不应依赖 Tauri 或具体数据库；同时，状态更新不能在持久化失败后只修改内存，否则 UI 会展示无法恢复的“幽灵状态”。未来的数据结构还需要支持迁移、降级保护和故障恢复。

## 决策

运行时采用三层边界：`nimora-runtime-core` 保存纯领域模型与不变量，`nimora-runtime-app` 提供用例和 `PetRepository` 端口，`nimora-persistence-sqlite` 实现 SQLite 适配器。桌面壳只负责组合这些层，不直接编写领域存储逻辑。

宠物和 Profile 当前事实分别保存为单例、版本化根快照；数据库行和 JSON 载荷都记录版本，载荷根对象使用 `schemaVersion`。首版数据库一次性创建宠物快照、Profile 快照、`event_outbox` 和用户程序权限表。状态快照与导致该状态的完整 Event 必须在同一个 SQLite 事务提交，并遵循“复制当前状态 → 修改候选状态 → 校验 → 创建 Command/Event → 原子提交快照与 Outbox → 发布到内存与事件缓冲区”的顺序。任一数据库写入失败都必须整体回滚，不得发布候选状态或事件。

SQLite 使用 bundled 构建以降低终端环境差异，启用外键、WAL、五秒 busy timeout 和 `synchronous=NORMAL`。应用拒绝读取高于自身支持范围的数据库或快照版本，避免旧版本静默破坏新数据。

## 版本与恢复

- 项目尚未发布，当前数据库只有首版 Schema；初始化必须在单个事务中一次完成。
- 当前明确拒绝非零且非首版的数据库版本，不维护开发期中间 Schema，也不猜测性迁移未知数据。
- 每次反序列化后必须重新执行领域校验，不能信任磁盘数据。
- 引入首个破坏性迁移前，必须实现迁移前备份、磁盘空间检查、失败回滚和只读安全模式。
- 当前数据库版本为 `1`。测试覆盖原子初始化、未知版本拒绝、快照恢复，以及 Event 主键冲突时快照写入回滚。真实稳定版发布后若需要升级，必须基于已发布 Schema 和真实数据样本设计迁移，不能复活开发期中间版本。
- `event_outbox` 使用 `pending`、`leased`、`delivered`、`dead-letter` 四态协议。消费者以合法稳定 ID 在 Immediate 事务中有界领取，领取会增加尝试次数并设置租约；租约过期可由其它消费者重领，旧消费者 ACK 或失败回报必须被拒绝。失败可设置下一次可用时间，达到最大尝试次数后进入死信；仅已确认记录可按时间和批量上限清理。容量为 256 的内存事件总线继续服务即时 UI，不承担可靠投递。
- Control Center 只读取各状态计数，不读取事件正文、租约所有者或错误详情。具体 Connector 必须提供稳定幂等键并自行选择重试策略；在真实消费者和端到端投递测试落地前，不宣称已经提供外部 at-least-once 投递，更不得声称 exactly-once。
- `SqlitePetRepository::backup_to` 与 `SqliteProfileRepository::backup_to` 使用 SQLite Online Backup API 创建一致性副本，覆盖 WAL 页面；禁止用普通文件复制替代。当前已验证备份副本可重新打开并恢复最新宠物快照，但自动迁移前调度、空间预检和只读恢复模式仍未完成。
- 将来如需要事件历史，事件日志只能作为审计或派生来源，不能取代状态快照作为唯一事实。

## 结果

- 正面：领域与数据库解耦，离线重启可恢复，失败写入不会污染内存状态，状态变化不会在数据库中丢失对应事件，后续可替换或扩展存储适配器。
- 代价：快照载荷与数据库 Schema 各自需要版本治理；频繁窗口移动可能产生较多写入。
- 缓解：后续为原生移动事件增加合并写入，并在第一个升级迁移前完成备份与安全模式门禁。
