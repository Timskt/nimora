# ADR-008：宠物运行状态采用版本化 SQLite 快照

> 状态：Accepted
> 日期：2026-07-17

## 背景

AsterPet 必须在完全离线时保存宠物状态，并在应用重启后可靠恢复。领域核心不应依赖 Tauri 或具体数据库；同时，状态更新不能在持久化失败后只修改内存，否则 UI 会展示无法恢复的“幽灵状态”。未来的数据结构还需要支持迁移、降级保护和故障恢复。

## 决策

运行时采用三层边界：`asterpet-runtime-core` 保存纯领域模型与不变量，`asterpet-runtime-app` 提供用例和 `PetRepository` 端口，`asterpet-persistence-sqlite` 实现 SQLite 适配器。桌面壳只负责组合这些层，不直接编写领域存储逻辑。

宠物当前事实保存为单例、版本化根快照；数据库行和 JSON 载荷都记录版本，载荷根对象使用 `schemaVersion`。写入在 SQLite 事务中完成，并遵循“复制当前状态 → 修改候选状态 → 校验 → 创建 Command → 提交持久化 → 发布到内存”的顺序。持久化失败时不得发布候选状态。

SQLite 使用 bundled 构建以降低终端环境差异，启用外键、WAL、五秒 busy timeout 和 `synchronous=NORMAL`。应用拒绝读取高于自身支持范围的数据库或快照版本，避免旧版本静默破坏新数据。

## 迁移与恢复

- 数据库迁移必须按版本递增、事务化且可重复执行。
- 每次反序列化后必须重新执行领域校验，不能信任磁盘数据。
- 引入首个破坏性迁移前，必须实现迁移前备份、磁盘空间检查、失败回滚和只读安全模式。
- 当前数据库仅有初始版本 `1`，尚不存在破坏性迁移；自动迁移前备份不属于本 ADR 对当前实现的完成声明。
- 将来如需要事件历史，事件日志只能作为审计或派生来源，不能取代状态快照作为唯一事实。

## 结果

- 正面：领域与数据库解耦，离线重启可恢复，失败写入不会污染内存状态，后续可替换或扩展存储适配器。
- 代价：快照载荷与数据库 Schema 各自需要版本治理；频繁窗口移动可能产生较多写入。
- 缓解：后续为原生移动事件增加合并写入，并在第一个升级迁移前完成备份与安全模式门禁。
